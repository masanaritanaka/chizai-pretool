## 1. プロダクト概要

- **何**: 特許・実用新案・意匠・商標の初期調査と構造化を支援する、デスクトップ／Webアプリ。
- **ポジション**: 弁理士・知財部へ渡す前の「前処理ツール」。非専門家（事業担当・エンジニア・デザイナー）が自分で論点整理できる状態にする。
- **やらないこと（非目標）**: 登録可能性の判定、法的助言、J-PlatPat の置き換え、特許DBの自前構築。
- **免責**: すべての調査系出力に「登録可能性を保証しない／初期調査・論点整理の補助」を常設表示する。

## 2. 設計原則：10 → 3（最重要）

入口は10個のプリセット（ボタン）だが、**10個を独立機能として実装してはならない。** 中身は3エンジンで、プリセットは「エンジン × 法域 × 出力テンプレ」の組み合わせにすぎない。

| # | プリセット | クラスタ | エンジン |
|---|-----------|---------|---------|
| 1 | 商標ネーミング危険度チェッカー | 調べる | Research |
| 2 | （商標系の派生・別テンプレ） | 調べる | Research |
| 3 | 特許の素人向け翻訳 | 調べる | Research |
| 4 | 自社アイデアの先行技術メモ化 | 調べる | Research |
| 5 | 意匠・UI類似チェック（画像入力） | 調べる | Research + Vision |
| 9 | 契約・提案書の知財リスクチェッカー | 調べる | Research |
| 7 | 競合知財ウォッチャー | 監視する | Watch（Phase 2・簡易版） |
| 8 | 特許マップ自動生成 | 監視する | Watch（Phase 2・簡易版） |
| 6 | 知財期限・ステータス管理 | 管理する | Manage |
| 10 | 防衛公開メモ | 管理する | Manage |

プリセットは `src/home/presets.ts` に**メタデータとして1か所**で定義する（engine, 法域, 入力タイプ, 出力テンプレ, 表示名, クラスタ）。プリセット追加＝この配列に1要素足すだけ、を構造的に保証すること。

## 3. 3コアエンジン

### 3.1 Research / Compose エンジン（presets 1, 2, 3, 4, 5, 9）
固定パイプライン：**入力 → 構造化 → 検索式＋論点 → 出力**。

- preset 間の差分は「法域セレクタ」「出力テンプレ」「入力タイプ（テキスト/番号/URL/画像）」のみ。ロジック本体は共通。
- 出力に必ず含めるもの：構造化メモ（技術分野・課題・解決手段・構成要素 等、法域に応じて変形）、類似語・英語表現への展開、**J-PlatPat 検索式とディープリンクURL**、危険度／論点の所感、「専門家に確認すべき論点」。
- preset 5 のみ画像入力（Claude Vision）を追加。形は同じ。

### 3.2 Watch エンジン（presets 7, 8）
- **Phase 2。MVP では作り込まない。** 当面は Research の出力を流用し「監視したい対象の検索式を生成して J-PlatPat ディープリンクで開く」簡易版のみ。
- 定期取得・コーパス構築・スクレイピングは**実装しない**（§4参照）。

### 3.3 Manage エンジン（presets 6, 10）
- ローカル SQLite の CRUD。**AI 不要。**
- 6: 出願日・公開日・登録日・審査請求期限・年金/更新期限・応答期限などの期限管理＋通知。
- 10: 防衛公開メモ（アイデア記録＋タイムスタンプ＋公開/非公開管理）。

## 4. データソース方針（実装前に必読・厳守）

- 特許庁「特許情報取得API」は **2024-08-09 で新規申込受付を終了**、ブラウザ（JS）からの直接アクセス不可、レート制限が厳しい。→ **使わない。**
- J-PlatPat のスクレイピングは ToS 上問題があり脆弱。→ **禁止。**
- 特許情報標準データ（バルク）での自前DB構築は重量級で配布アプリに同梱不可。→ **やらない。**
- **唯一のオンライン依存は Claude API。** 「検索」はアプリが行わず、Claude が J-PlatPat 検索式を生成し、ユーザーが J-PlatPat 上で実行する（ディープリンクで遷移）。
- → Claude Code は特許庁API・各庁API・スクレイピングを**勝手に実装しないこと。** 検索が必要な場面は必ず「検索式生成＋J-PlatPatディープリンク」で解決する。

## 5. 技術スタック

- フロント: Vite + React + TypeScript
- デスクトップ: **Tauri 2**（Windows: `.msi`/`.exe`、macOS: `.dmg`）。軽量インストーラ。
- ローカルDB: SQLite（`tauri-plugin-sql`）— Manage エンジン用。
- APIキー: **BYO（ユーザー持ち込み）。** OSキーチェーン（`tauri-plugin-stronghold` もしくは OS keychain）に保存。**平文保存・リポジトリ同梱・フロント埋め込みは禁止。**
- Web版（任意・Phase で後送り可）: 同一フロントを Vercel/Railway にデプロイ。キーは薄いサーバープロキシ（環境変数）経由。フロントにキーを置かない。
- Claude API: `/v1/messages` エンドポイント。モデルは `claude-sonnet-4-6`（必要に応じ上位）。

## 6. 画面構成

- **Home**: 10ボタンを「調べる／監視する／管理する」の3クラスタに**視覚的にグルーピング**して配置（1・3・4 が触り心地的に混同されやすいので、クラスタ表示で混乱を消す）。各ボタンは `presets.ts` のメタを読むだけ。
- **エンジンページ（共有）**: preset に応じて法域セレクタ・入力フォーム・出力テンプレが切り替わる。Research は1ページの使い回し。
- **設定**: Claude APIキーの入力・保存・削除、免責表示。

## 8. 完成の定義（Definition of Done）— 「とりあえず動く」防止チェックリスト

各項目が満たされるまで「完了」と言わないこと。

- [ ] 第三者が GitHub Releases から Win/Mac のインストーラを入手し、起動できる
- [ ] 初回起動で Claude APIキーを入力 → OSキーチェーンに保存 → 再起動後も保持される
- [ ] Research エンジンが E2E で動く（入力 → 構造化 → 検索式 → J-PlatPat ディープリンクが開く）
- [ ] プリセットを1個追加するのに `presets.ts` への追記だけで済む構造になっている
- [ ] GitHub Actions が Win/Mac 両方のインストーラを自動生成し、Releases に添付する
- [ ] エラー処理：キー未設定／API失敗／オフライン／レート超過 を UI で明示する
- [ ] README にクイックスタート（インストール手順・キー設定・最初の操作）がある
- [ ] 免責表示（法律判断しない／初期調査補助）がアプリ内と全調査出力に常設されている
- [ ] LICENSE 設置済み、`.gitignore` 整備済み、キー・秘匿情報をコミットしていない
- [ ] 法域追加・テンプレ追加の手順が `CLAUDE.md` か `docs/` に1ページある
